# Zx50 Network ROM Emulator & Dual-Port RAM Daughterboard

## 1. Concept Overview
The goal is to replace the DIP32 SST39SF040 (512KB Flash ROM) on the Zx50 Memory Card with a custom Wi-Fi-enabled microcontroller daughterboard. 

By mapping the microcontroller directly into the Z80's memory or I/O space, it acts as a **Hardware ROM Emulator** and a "Network File System." This allows for over-the-air (OTA) firmware updates, network booting, and dropping compiled `.bin` files directly into the Z80's memory space from a modern PC without pulling chips.

## 2. Microcontroller Selection
An 8 MHz Z80 asserts `~CE` (Chip Enable) and `~OE` (Output Enable) for roughly 150–200 nanoseconds. Standard software interrupts on modern microcontrollers are too slow (1–2 µs) to fetch data and drive the bus in time. We must use hardware-accelerated parallel I/O.

**Option A: The RP2040 (Raspberry Pi Pico architecture)**
* **The Magic:** Features Programmable I/O (PIO). A tiny PIO state machine can monitor `~CE`/`~OE`, read the address bus, and output the data byte in 10–20 ns, easily mimicking a 70ns SRAM.
* **The Catch (Pin Count):** A full 512KB DIP32 replacement requires 30 GPIO pins (19 Address, 8 Data, 3 Control). A standard Pico W only exposes 26.
* **Workarounds:**
  1. Emulate a 32KB ROM (A0-A14) to fit within 26 pins.
  2. Use an I/O Port trap (`IN`/`OUT`) instead of flat memory mapping.
  3. Design a custom PCB using the bare RP2040 QFN-56 chip (which has 30 GPIOs) paired with an ESP8285 for Wi-Fi.

**Option B: The ESP32-S3**
* **The Magic:** Native Wi-Fi built into the silicon, up to 45 GPIO pins, and abundant internal RAM/Flash.
* **Bus Speed:** Features a "Dedicated GPIO" subsystem and parallel LCD/Camera interfaces that can be abused to handle 8-bit bus transactions fast enough for the Z80.
* **The Verdict:** The best choice if designing a custom PCB from scratch, as it requires fewer external components than a bare RP2040 + Wi-Fi combo.

## 3. The "5V Elephant" (Level Shifting)
The Z80 backplane operates at 5V, while the RP2040/ESP32 are strictly 3.3V and are **NOT 5V tolerant**. Plugging them directly into the DIP32 socket will destroy them.

* **Address & Control Lines (Z80 -> MCU):** Use unidirectional 5V-to-3.3V buffers (e.g., `74LVC541` or `74AHC541`).
* **Data Bus (Bidirectional):** Use a high-speed 8-bit transceiver (e.g., `74LVC245` or `74ALVC245`).
  * The Z80 `~RD` (Read) pin must be tied to the transceiver's `DIR` (Direction) pin so it only drives 3.3V logic *towards* the Z80 when the CPU is reading, and accepts 5V logic *towards* the MCU when the CPU is writing.

## 4. Physical Layout: The "Mushroom" Geometry
You cannot fit an MCU, Flash, Wi-Fi module, and five level-shifting ICs into a standard 1.6" x 0.6" DIP32 footprint. The daughterboard must use a "Mushroom" or "Cantilever" design.

* **The Stem:** Two rows of standard 2.54mm male headers on the bottom to plug into the Zx50's D3 socket.
* **The Core:** Level-shifting ICs are placed immediately above the pins to keep the 5V traces as short as possible. The MCU sits in the center of the expanded upper board.
* **The Overhang (RF Constraints):** The Wi-Fi antenna must be placed at the absolute "North" edge of the daughterboard (hanging over the D3/R3 silkscreen area of the main board). 
* **Noise Avoidance:** By extending the top of the daughterboard slightly above the main Zx50 card, the antenna gains a clear line-of-sight, escaping the localized electromagnetic noise generated by the 5V 74ABT transceivers and the Atmel CPLD.
* **Z-Height:** To avoid crashing into adjacent backplane cards, the daughterboard must use low-profile DIP pins and exclusively surface-mounted components.
